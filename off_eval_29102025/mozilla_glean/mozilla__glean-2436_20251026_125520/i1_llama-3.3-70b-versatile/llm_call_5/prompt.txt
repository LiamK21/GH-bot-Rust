Issue:
<issue>
HTTP `Date` header set at ping submission time, not when uploaded - does that meet expectations?
We [set the `Date` header of the HTTP request](https://github.com/mozilla/glean/blob/4525045599c96d23f74e10b269299d62549facdc/glean-core/src/upload/request.rs#L71) as [per the HTTP edge spec](https://docs.telemetry.mozilla.org/concepts/pipeline/http_edge_spec.html#supported-http-headers).

In Glean this is [done when enqueuing a ping](https://github.com/mozilla/glean/blob/main/glean-core/src/upload/mod.rs#L384).

Enqueuing happens in two cases:

* The app is running and a ping is submitted
* The app launches and pending pings are loaded from disk

In both cases the data is not sent immediately, and depending on circumstances the actual HTTP request can be some seconds, minutes or, according to our data, hours in the future.
That makes this value effectively useless for any skew calculation.

Now to my knowledge no one really depends on this value.
Everyone just uses the `submission_timestamp` anyway (the server's time when ingesting the ping).

We could still consider changing this value and only setting the date header before handing out the request to the uploader.
</issue>
